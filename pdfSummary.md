# 비동기 엑셀 생성 시스템 요구사항

## 1. 개요

사용자가 엑셀 생성 요청을 하면 API는 즉시 응답을 반환하고, 실제 엑셀 파일 생성은 백그라운드에서 비동기로 처리하는 시스템을 구현한다.

- 대상 데이터: 주문 데이터 최소 100,000건
- 엑셀 생성 방식: 백그라운드 비동기 처리
- 상태 관리 단위: Job
- 실행 환경: Docker Compose
- WAS 제약: 단일 컨테이너, 고정 리소스, 스케일아웃 불가
- 파일 저장소: 컨테이너 로컬 볼륨

---

## 2. 실행 환경 요구사항

### 2.1 실행 방식

전체 서비스는 아래 명령어 하나로 실행 가능해야 한다.

```bash
docker compose up --build
```

### 2.2 Docker Compose 필수 구성

Docker Compose 환경에는 다음 구성이 포함되어야 한다.

- Backend API 서버
- Frontend 서버
- Database
- 파일 저장용 로컬 볼륨
- DB 시드 데이터 자동 삽입 스크립트

### 2.3 WAS 리소스 제한

WAS는 단일 컨테이너로만 실행한다.

- 스케일아웃 허용하지 않음
- 고정 리소스 환경에서 동작해야 함

```yaml
deploy:
  resources:
    limits:
      cpus: '0.5'
      memory: 1G
    reservations:
      cpus: '0.25'
      memory: 512M
```

## 3. 데이터베이스 요구사항

### 3.1 주문 데이터 테이블

아래 스키마를 기준으로 주문 데이터 테이블을 생성한다.

| 컬럼명       | 타입      | 설명                 |
| ------------ | --------- | -------------------- |
| id           | INTEGER   | PK, 자동 증가 기본키 |
| user_name    | VARCHAR   | 주문자 이름          |
| product_name | VARCHAR   | 상품명               |
| category     | VARCHAR   | 카테고리             |
| amount       | INTEGER   | 결제금액, 원 단위    |
| status       | VARCHAR   | 주문상태             |
| order_date   | TIMESTAMP | 주문일시             |

### 3.2 주문 상태 값

`status` 컬럼은 아래 값 중 하나를 가진다.

- `confirmed`
- `cancelled`
- `pending`

### 3.3 카테고리 예시

`category` 컬럼은 아래와 같은 값을 사용할 수 있다.

- 스키
- 워터파크
- 숙박
- 기타 자유 구성 가능

### 3.4 시드 데이터

서비스 실행 시 최소 100,000건 이상의 더미 주문 데이터를 자동 삽입해야 한다.

- Docker Compose 실행 시 자동 생성
- 중복 실행 시에도 안정적으로 처리 가능해야 함
- 데이터 생성 방식은 자유 선택

## 4. 백엔드 요구사항

### 4.1 엑셀 생성 요청

백엔드는 엑셀 생성 요청 API를 제공해야 한다.

요구사항:

- 요청을 받으면 즉시 응답 반환
- 실제 엑셀 생성은 백그라운드에서 처리
- 응답에는 생성된 Job 식별자 포함

예시 응답:

```json
{
  "job_id": 1,
  "status": "pending"
}
```

### 4.2 비동기 처리

엑셀 생성 작업은 Job 단위로 비동기 처리한다.

요구사항:

- API 요청 스레드에서 직접 엑셀을 생성하지 않음
- 백그라운드 워커 또는 내부 큐 기반 처리
- 단일 WAS 컨테이너 환경에서 안정적으로 동작
- 100,000건 데이터 처리 가능

### 4.3 Job 상태 관리

Job은 아래 상태를 가진다.

| 상태       | 설명              |
| ---------- | ----------------- |
| pending    | 생성 요청 접수됨  |
| processing | 엑셀 생성 처리 중 |
| done       | 엑셀 생성 완료    |
| failed     | 엑셀 생성 실패    |

### 4.4 Job 메타데이터 관리

Job 테이블에는 최소 아래 정보를 저장해야 한다.

| 컬럼          | 설명                       |
| ------------- | -------------------------- |
| job_id        | Job 식별자                 |
| status        | Job 상태                   |
| requested_at  | 요청일시                   |
| started_at    | 작업 시작 시간             |
| completed_at  | 작업 완료 시간             |
| file_path     | 생성된 엑셀 파일 저장 경로 |
| error_message | 실패 시 에러 메시지        |

### 4.5 파일 저장

엑셀 생성 완료 후 파일을 컨테이너 로컬 볼륨에 저장한다.

요구사항:

- 파일 저장 경로를 DB에 기록
- 컨테이너 재시작 후에도 파일 유지 가능하도록 볼륨 사용
- 파일명 중복 방지 필요

예시 파일명:

```text
orders_export_job_1.xlsx
```

### 4.6 엑셀 생성 라이브러리

엑셀 생성 라이브러리는 자유롭게 선택할 수 있다.

예시:

- Java: Apache POI, SXSSF
- Node.js: ExcelJS
- Python: openpyxl, xlsxwriter

단, 100,000건 데이터를 메모리 제한 내에서 처리할 수 있어야 한다.

### 4.7 API 설계

API 구조 및 엔드포인트 설계는 지원자 자율이다.

예시 API:

| Method | Endpoint                            | 설명                      |
| ------ | ----------------------------------- | ------------------------- |
| POST   | `/api/export-jobs`                  | 엑셀 생성 Job 요청        |
| GET    | `/api/export-jobs`                  | Job 목록 조회             |
| GET    | `/api/export-jobs/{jobId}`          | Job 상세 조회             |
| GET    | `/api/export-jobs/{jobId}/download` | 완료된 엑셀 파일 다운로드 |

## 5. 프론트엔드 요구사항

### 5.1 데이터 요청 목록 페이지

프론트엔드는 데이터 요청 목록 페이지를 구현한다.

화면에는 Job 목록을 테이블 형식으로 표시한다.

| 컬럼     | 설명                                 |
| -------- | ------------------------------------ |
| job_id   | Job 식별자                           |
| 요청일시 | 엑셀 생성 요청 시간                  |
| 시작시간 | 엑셀 생성 시작 시간                  |
| 완료시간 | 엑셀 생성 완료 시간                  |
| 상태     | pending / processing / done / failed |
| 파일경로 | 생성된 파일 경로                     |

### 5.2 엑셀 생성 요청 버튼

데이터 요청 목록 페이지에는 엑셀 생성 요청 버튼이 있어야 한다.

버튼 클릭 시:

1. 엑셀 생성 요청 API 호출
2. Job 생성
3. API는 즉시 응답
4. 목록에 신규 Job 반영

### 5.3 상태 확인 방식

엑셀 생성 상태 확인 방식은 아래 중 하나를 선택하여 구현한다.

- 페이지 새로고침
- 주기적 폴링
- SSE(Server-Sent Events)

권장 방식:

```text
주기적 폴링
```

예시:

- 2~3초마다 Job 목록 조회
- 상태가 `done` 또는 `failed`인 Job은 최종 상태로 표시
- `done` 상태일 경우 다운로드 가능 처리

## 6. 구현 제약사항

### 6.1 스케일아웃 금지

WAS는 단일 컨테이너로만 실행해야 한다.

허용하지 않는 구성:

- WAS 컨테이너 다중 실행
- 외부 분산 큐 기반 스케일아웃
- Kubernetes 기반 수평 확장

### 6.2 리소스 제한 고려

아래 리소스 제한 내에서 정상 동작해야 한다.

- CPU limit: `0.5`
- Memory limit: `1G`
- CPU reservation: `0.25`
- Memory reservation: `512M`

### 6.3 대용량 엑셀 생성 고려

100,000건 이상의 데이터를 엑셀로 생성해야 하므로 아래 사항을 고려해야 한다.

- 전체 데이터를 한 번에 메모리에 적재하지 않기
- DB 조회 시 페이징 또는 스트리밍 처리
- 엑셀 작성 시 스트리밍 방식 사용 권장
- 백그라운드 작업 동시 실행 개수 제한 필요
- 실패 시 Job 상태를 `failed`로 변경
